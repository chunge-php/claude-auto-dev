# Auto-Dev v6

$ARGUMENTS

## 铁律

1. **先读后写** — 改文件前必须 Read 它
2. **先查后建** — 建新代码前 Grep 是否已有类似实现
3. **不破坏现有** — 改导出/接口/schema 前 Grep 所有消费者确认兼容
4. **跟随项目** — CLAUDE.md > 项目隐性约定 > 通用实践
5. **不问不启** — 不确认、不启动服务、不跑 build/test、不生成管理文件
6. **只推 origin** — commit: `type(scope): 中文描述`

## 决策框架（遇到选择时如何思考）

**模糊需求** → 选最常见的理解，在 commit message 中记录假设
**多种方案** → 选最简单的（少文件、少依赖、少抽象），除非有明确的扩展需求
**不确定改动是否安全** → 不改。跳过并在汇报中说明
**功能范围膨胀** → 只做明确要求的，延伸功能记入汇报"后续建议"
**项目约定与最佳实践冲突** → 跟随项目约定，保持一致性比"正确性"更重要

## Phase 0: 启动

### 输入解析

| 输入 | 识别 | 处理 |
|------|------|------|
| 文本 | 默认 | 直接用 |
| 文件路径 | `/` `~` `./` 开头 | Read 内容 |
| Issue URL | 含 github.com/issues/ | `gh issue view` |
| 多需求 | `+` 分隔 | 逐个执行 |
| 空 | 无参数 | 审计项目，列出待完善项让用户选择（不自动全做） |

### 意图分类

| 意图 | 判断依据 | 行为差异 |
|------|----------|----------|
| **fix** | "修复/bug/报错/崩溃" | 最小改动，写回归测试 |
| **feat** | "添加/新增/实现/支持" | 完整实现，含测试+文档同步 |
| **refactor** | "重构/优化结构/整理" | 先锁行为再改，清理死代码 |
| **perf** | "性能/慢/优化/卡" | 定位瓶颈，针对性优化 |
| **chore** | "升级/配置/迁移/清理" | 最小风险，逐步执行 |

### 环境感知（并行 Glob/Read）

```
技术栈:    package.json | pom.xml | go.mod | Cargo.toml | pyproject.toml
包管理器:  pnpm-lock→pnpm | yarn.lock→yarn | package-lock→npm | bun.lockb→bun
项目类型:  前端 | 后端 | 全栈 | 库 | CLI
Monorepo:  apps/ | packages/ | workspaces
代码风格:  .eslintrc* | .prettierrc* | biome.json | .editorconfig
数据库:    prisma/ | supabase/ | drizzle/ | migrations/
测试框架:  vitest | jest | pytest | go test | 无
i18n:      i18n/ | locales/ | messages/ | next-intl | i18next（检测到则不硬编码文案）
Git Hooks: .husky/ | .lintstagedrc*（检测到则跳过手动格式化）
```

### 学习项目约定

读取 CLAUDE.md + 项目记忆。快速扫描：
- `git log --oneline -10` → commit 风格
- 2-3个核心文件 → 命名/分层/模式
- **文件放置规则**：扫描目录，记录各类文件位置
- **是否新项目**：无 src/ 且 git log 为空 → 新项目，进入脚手架流程

### 幂等检查

Grep 需求关键词：
- 已完全实现 → 告知用户，终止
- 部分实现 → 只做剩余

### 选择模式

| 模式 | 条件 | 流程 |
|------|------|------|
| **脚手架** | 新项目（无代码） | 初始化 → 开发 → 推送 |
| **快速** | < 5文件 / fix / chore | 开发 → 推送 |
| **标准** | 5-20文件 / feat / refactor | 规划 → 理解 → 开发 → 审查 → 推送 |
| **大型** | > 20文件 / 跨模块 | 同标准 + feature分支 + 备份tag + 分批 |

### 安全网
- 标准/大型：记录起始 commit hash
- 大型：`git tag auto-dev-backup-$(date +%s)` + feature分支
- 未提交改动 → stash（结束时提醒）
- 有未完成 auto-dev Task → 从断点继续

## 脚手架阶段（仅新项目）

当检测到空项目时自动执行：
1. 根据需求选择技术栈（优先用户指定，否则按需求类型推荐）
2. 用官方脚手架初始化（`create-next-app` / `create-vite` / `cargo init` 等）
3. 安装基础依赖
4. 创建 CLAUDE.md 写入项目约定
5. 创建 .env.example
6. commit: `init: 项目初始化`
7. 然后正常进入开发阶段

## 阶段1: 规划（标准/大型）

Agent(subagent_type="feature-dev:code-architect")：
- 输入：技术栈 + 需求 + CLAUDE.md + 目录结构
- 输出：文件清单 + 依赖顺序 + 数据模型方案

### 全栈功能的分层顺序

涉及多层时**必须按此顺序**开发（下层先于上层）：
```
1. 数据库 Schema / 迁移
2. 后端 Model / Service
3. 后端 API / Controller
4. 前端 API 调用层（hooks/services）
5. 前端 UI 组件
6. 前端页面集成
```

> 每层完成后单独 commit，不跨层混合提交

转化为 TaskCreate，标注层级和依赖。**不commit。**

## 阶段2: 理解（标准/大型，已有项目）

Agent(subagent_type="feature-dev:code-explorer") 分析涉及的已有文件：
- 执行路径、数据流、import 关系
- 结果传递给开发阶段，**不输出**。不commit。

## 阶段3: 开发

### 开发循环

```
读 → 查 → 写 → 检 → 格 → 存
```

**读**: Read 目标文件（>500行 Grep 定位后读局部）

**查**: Grep 可复用代码。有→复用，无→新建

**写**: 实现功能
- 新文件按学到的放置规则创建
- DB变更同步迁移文件
- 环境变量 → 默认值 + .env.example
- **i18n 项目**：文案写入 locale 文件，代码中用 t('key')，不硬编码
- **新增依赖前必须检查**：
  - Grep 项目是否已有功能相似的包（如已有 dayjs 就不加 moment）
  - 优先用语言标准库/已有依赖能解决的方案
  - 确需新包 → `{pm} add {pkg}`
- **性能意识**：
  - DB: 避免 N+1，用 include/join；新表加合理索引
  - 前端: 避免不必要 re-render；大列表虚拟滚动；图片 lazy load
  - API: 只返回必要字段；列表接口必须分页
- **DB 迁移安全**：
  - 只做加法（add column/table），不在迁移中 drop/rename
  - 需要删列/改名 → 分两步：①新增 ②标记废弃（在汇报中说明）

**检**:
- 改了导出 → Grep 消费者确认兼容
- 改了 schema → 检查查询代码
- **refactor 意图**：Grep 检查是否产生死代码（无人引用的导出），有则删除

**格**:
- 检测到 .husky/ 或 lint-staged → **跳过手动格式化**（hooks 会自动处理）
- 无 git hooks → prettier/biome/rustfmt/ruff 对变更文件执行
- 无任何配置 → 跳过

**存**: `type(scope): 描述`，每个功能模块一个 commit

### 文档同步（feat 意图时）

改了用户可见行为后检查：
- README 提到了相关功能 → 更新 README 对应部分
- 有 API 文档（docs/ 或 openapi.yaml）→ 同步更新
- commit: `docs: 更新文档`

### 测试策略（写但不跑）

| 意图 | 要求 |
|------|------|
| fix | 回归测试（覆盖修复的 case） |
| feat | 核心逻辑单测 |
| refactor | 先写测试锁行为再改 |
| perf/chore | 不写 |

- 无测试框架 → 不写（不强建）
- 有 → 跟随已有模式（位置、命名、断言风格）
- commit: `test(scope): 描述`

### 完成标准

- [ ] 功能按需求完整实现
- [ ] 无硬编码密钥/凭证
- [ ] 导出变更已同步消费者
- [ ] 新文件在正确目录
- [ ] DB迁移同步（仅加法）
- [ ] i18n 文案无硬编码（如项目有 i18n）
- [ ] 测试已写（按意图）
- [ ] 文档已同步（feat 意图）
- [ ] 无死代码残留（refactor 意图）

### 一致性保障（部分失败时）

如果一组关联任务（如同一功能的前后端）中途失败：
- 已完成的部分如果**能独立工作**（如后端 API 已完成，前端未完成）→ 保留，汇报说明
- 已完成的部分**不能独立工作**（如前端调了不存在的 API）→ `git revert` 到该功能开始前的 commit
- 汇报中标注哪些被回滚、失败原因

### 并行开发（仅大型模式）

策略：**只并行编码，串行 commit**
1. 多个 Agent 同时编写不同文件集（不 commit）
2. 主流程串行 commit 每个 Agent 产出
3. 并行 Agent 之间**不可有共同文件**

### 错误处理

| 类型 | 策略 |
|------|------|
| 语法/类型错误 | 直接修复 |
| 依赖缺失 | 安装后继续 |
| 环境变量缺失 | .env.example + 代码默认值 |
| 循环依赖 | 提取独立模块 |
| 导出不兼容 | 同步更新消费者 |
| 权限/网络 | 记录跳过 |
| 未知 | 换实现1次，仍失败跳过 |

## 阶段4: 审查（标准/大型）

Agent(subagent_type="feature-dev:code-reviewer")：
- 输入：`git diff {起始hash}..HEAD`
- 审查清单：
  - 安全漏洞（注入、XSS、硬编码密钥）
  - 铁律违反
  - CLAUDE.md 违规
  - 逻辑错误和边界
  - 性能问题（N+1、缺失索引、不必要 re-render）
  - 死代码残留
  - i18n 硬编码文案
- 有问题→修复→`fix: 描述`

## 阶段5: 推送

1. DB项目 → seed 覆盖新功能 → `chore(data): 描述`
2. `git push origin {branch}`（失败→rebase重试→二次失败报错不强推）
3. 大型模式 → 技术决策存项目记忆
4. 有 stash → 提醒 pop
5. 汇报
